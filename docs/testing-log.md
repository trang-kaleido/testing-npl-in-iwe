## 2026-06-24 (later) — closed-class GECToR tagger implemented; all three "still open" items resolved

**Checkpoint**: `gotutiyan/gector-roberta-base-5k`, installed via `pip install git+https://github.com/gotutiyan/gector` (unofficial PyTorch GECToR, HF-hosted weights). Required raising `backend/.venv` from Python 3.9.6 to 3.12 (gector needs ≥3.11) and pinning `transformers==4.49.0` — newer `transformers` loads encoders through a meta-device path `gector.GECToR.from_pretrained()` doesn't handle. `torch` is now a direct dependency (was already present transitively via `sentence-transformers`).

**Offset mapping**: resolved by feeding the tagger spaCy's own tokenization (`is_split_into_words=True` with `[t.text for t in doc]`) rather than a custom word-splitter — each predicted tag aligns 1:1 with a spaCy token, so spans come straight from `token.idx` with no subword realignment. Confirmed empirically: for "She go to school yesterday because need book.", spaCy and the tagger produced the same 9-word alignment. Used `gector.predict.get_word_masks_from_word_ids` for the real word-mask (not an all-ones stand-in) and the library's own `process_token` to decode `$TRANSFORM_VERB_*`/`$REPLACE_*` literals — reuse over reimplementing the verb-form-vocab decode logic.

**Closed-class filter shipped** (`backend/main.py:_is_closed_class_tag`): `$KEEP`/`$DELETE`/`$MERGE_HYPHEN`/`$MERGE_SPACE` and all `$TRANSFORM_*` always allowed; `$APPEND_`/`$REPLACE_` allowed only for articles (`a`, `an`, `the`) and a 10-word preposition list. Verified the real model's tag vocabulary (5002 tags) actually contains exactly these as discrete tags (not synthesized) before shipping the filter.

**Latency measured** (real model, real LanguageTool via local `erikvl87/languagetool` Docker image, M-series CPU, no GPU): first call after process start ~1.3s (one-time warm-up); subsequent calls ~100–165ms total for `/accuracy` (LT + tagger run concurrently via `asyncio.gather`/`asyncio.to_thread`). No hard budget was pinned going in — this is within what feels acceptable for a sentence-terminator-triggered check, not a per-keystroke one.

**Verified merge behavior end-to-end**: "He live in house near park." — LT flags "live" (GRAMMAR, verb form) and the tagger's own verb-inflection hit on the same span is correctly dropped as an overlap; the tagger's two article insertions ("in [a] house", "near [the] park") survive since they don't overlap any LT match. "She is interested at music." surfaced a tagger-only closed-class preposition fix ("at"→"in"), a gap LT does not catch — confirms the gap-closing intent from the 2026-06-18/19 entries.

**Loose end**: SVA misses persist for some cases the model doesn't confidently tag (e.g. "They was happy" was not corrected) — expected, this is the base-size checkpoint; not pursuing further for spike purposes.

## 2026-06-24 — discriminative GEC boundary resolved; integration shape decided for accuracy

**Decision — closed-class tags only**: the 2026-06-19 entry's open question (does GECToR-style tagging count as "generation" under ADR 0001?) is resolved as ADR 0002: the line is drawn inside the tag vocabulary itself. Closed-class function-word/morphology tags (article insertion, a short preposition list, verb inflection — on the order of dozens, fixed at training time, same shape as `_COHESION_LINKERS`) are permitted as bounded analysis. `REPLACE_X→Y` lexical tags, which scale with vocabulary size into the thousands, are excluded as effectively open-vocabulary generation.

**Decision — parallel merge, not a sanitize-then-deep-check pipeline**: considered running LT first to "clean" the sentence before handing it to the tagger (the rationale being that GEC taggers are sensitive to noisy/garbled input). Rejected as overkill for this spike: accuracy has no corrected-document object to hand forward — both passes are advisory diagnostics over the same completed sentence — so sequencing would require actually applying one of LT's candidate replacements, picking among multiple candidates arbitrarily, then realigning the tagger's offsets through that rewrite back onto the original text. IELTS practice sentences are typically well-formed prose with isolated errors, not the kind of noise that meaningfully degrades tagger attention, so the realignment cost isn't bought by a real accuracy gain. Instead: run LT and the closed-class tagger independently on the original sentence, concatenate diagnostics, and drop any tagger hit whose span overlaps an existing LT match (LT wins on overlap, since it's the more explainable/precise source). This targets exactly the gaps named in the 2026-06-18 entry (missing article, subject-verb agreement) without touching LT's existing `/accuracy` reshaping logic (`backend/main.py:202-227`) beyond appending a second source of diagnostics.

**Still open**: which actual tagger checkpoint to use/host, subword-to-character offset mapping for the tagger's spans, and latency budget for a second model call per accuracy check.

## 2026-06-19 — recurring closed-vocabulary weakness; discriminative GEC as a possible answer for accuracy

**Bug found — sentence-boundary detection is a naive regex, not real segmentation**: both `frontend/src/stall.js` and `frontend/src/ghost.js` find the "current sentence" with `TERMINATOR_RE = /[.?!]/`, scanning backward to the last match. This breaks on abbreviations, decimals, and quoted terminators, and is duplicated independently in both files. Notably, spaCy is already loaded server-side for `/stall` and has a real sentencizer (`doc.sents`), but the boundary is decided client-side via regex before any NLP ever sees the text — so a wrong boundary can feed the wrong span into stall's parse. Filed as issue #8.

**Insight — the same architectural weakness recurs across stall, ghost text, and accuracy**: in every feature, a real analysis mechanism (spaCy's dependency parse, the ghost-text trie, LanguageTool's pattern engine) correctly diagnoses *that* something applies, then hands back a result from a small, static, closed vocabulary — `_VERB_OBJECTS` / `_WORD_COLLOCATES` / `_COHESION_LINKERS` in `backend/main.py` (hand-typed), and the original ghost-text trie before issue #6 (also hand-typed). Issue #6 already proved a fix for this shape: replace the hand-typed list with one mined from a real corpus (STREUSLE), keeping the same trie mechanism but widening its source data. The same move is open for stall's tables, and for using SBERT (currently just a tie-breaking re-ranker) as an open-vocabulary nearest-neighbor retrieval index instead.

**Note for future investigation — discriminative (non-generative) GEC for the accuracy gaps**: the 2026-06-18 entry below names two gaps LanguageTool's pattern rules don't cover — missing article ("use car") and subject-verb agreement ("air get polluted"). Both are *bounded* edit operations (insert a function word, change a verb form), not open rewrites. Grammarly's own published technique for exactly this, **GECToR** ("Grammatical Error Correction: Tag, Not Rewrite," Omelianchuk et al. 2020), is a sequence *tagger*, not a decoder: a transformer encoder classifies each token into a fixed edit-tag vocabulary (`KEEP`, `DELETE`, `APPEND_the`, `REPLACE_is→are`, verb-inflection tags, etc.), and a deterministic post-process applies the winning tag over a few iterative passes — no autoregressive generation. This is structurally closer to ADR 0001's "analyze, don't generate" boundary than a generative GEC model would be, since the model's output is a per-token label from a finite set, not freely composed text — though the tag set itself still bakes in specific words (e.g. `APPEND_the`), so whether token-tagging counts as "generation" under ADR 0001 is a real open question worth deciding deliberately before any implementation, not assuming either way.

**Where this still has a ceiling**: GECToR-style tagging only reaches errors inside its fixed tag vocabulary (function words, morphology) — open-vocabulary errors (wrong word choice, awkward phrasing) are out of reach for the same reason a generative model would be needed for them in the first place. If pursued, the shape would be augment LT (cheap, explainable, narrow) with a local frozen tagger checkpoint for the structural gaps, plus a merge/rank step — not a replacement of either.

## 2026-06-18 — ghost text data source replaced (issue #6)

**Bug found**: the shipped trie (`FORCED_CONTINUATIONS` in `frontend/src/ghost.js`) was 14 hand-typed phrase→continuation pairs with no cited source — it only ever matched those exact 14 strings, so there was nothing generalizable to test. Reopened #6.

**Fix**: replaced it with a list extracted and empirically validated from the STREUSLE corpus (`scripts/extract_ghost_continuations.py`, see that file for the pinned commit and method) — a prefix only counts as "forced" if every occurrence of it anywhere in the corpus is followed by the same word, with at least 2 occurrences. Result: 6 entries (`as soon`→` as`, `in front`→` of`, `in the middle`→` of`, `according`→` to`, `in need`→` of`, `when it comes`→` to`), down from 14, but every entry is now corpus-validated rather than invented.

**Domain-fit gap (open)**: STREUSLE is hand-annotated Yelp reviews, not academic writing. Checked directly: none of the original IELTS-register phrases ("in terms of", "with regard to", "as opposed to", "a wide range of", etc.) occur in the corpus at all, except "regardless of" / "in spite of" (which didn't pass the ≥2-occurrence cardinality check). So the shipped list proves the extraction *method* works, but doesn't yet cover academic-register ghost text — that needs a different, academic-domain corpus with comparable MWE annotation, which wasn't found in this pass.

**Implemented**: edits that overlap an existing diagnostic's span (manual retyping or clicking a suggestion) now trigger an automatic recheck of that sentence, instead of only rechecking when the user retypes the sentence terminator. This closes the "sequential reveal" gap from the entry below: correcting `"Many person has problem with me."` to `"Many people has problem with me."` now immediately surfaces the newly-revealed `PEOPLE_VBZ` flag on "has", without waiting for the user to retype the period.

**Future scope — this does not fix LT's coverage gaps**: the recheck loop only helps when a fix changes the literal text enough to match a *different* existing LT rule. It does nothing for errors LT has no rule for at all, in any phrasing — e.g. the missing-article ("use car") and subject-verb-agreement ("air get polluted") misses noted below never get caught no matter how many times the sentence is rechecked, since there's no pattern to match in the first place. Closing that gap means changing the grammar-checking backend, not rechecking more.

**Note for future investigation**: Grammarly's pipeline is reportedly a hybrid — a layer of fast, narrow, rule/pattern-based checks (similar in spirit to LanguageTool: explainable, low-latency, high-precision) combined with a neural GEC (Grammar Error Correction) model that handles the broader, parsing-dependent corrections rule-based systems miss, with a ranking/merge step reconciling both sets of suggestions. Worth verifying this against primary sources before committing engineering effort, but it maps onto our two earlier options (swap vs. augment): an "augment" path would mean keeping LT for its cheap/explainable matches and adding a GEC model alongside it for the rest, rather than fully replacing LT.

## 2026-06-18 — accuracy squiggles (issues #1, #2)

**Setup**: LanguageTool sidecar (`erikvl87/languagetool` Docker image) + FastAPI backend + Vite frontend, run locally per `docs/dev-startup.md`.

**Bug found — squiggles didn't clear on fix**: editing the flagged word inside an already-terminated sentence (without retyping the sentence terminator) left the red squiggle in place. Root cause: `diagnosticField` in `frontend/src/editor.js` only ever repositioned diagnostics via `mapPos` on doc changes and merged new diagnostics additively — nothing ever invalidated a diagnostic once added, and a clean recheck (zero new matches) didn't dispatch at all. Fixed: edits overlapping a diagnostic's span now drop it immediately, and a recheck replaces (rather than merges with) prior diagnostics in its sentence range, including the zero-match case.

**Insight — LanguageTool's rule coverage is pattern-based, not general grammar parsing**: tested `"For example when too many person use car, air get polluted."` — only the "many person" agreement error was flagged; "use car" (missing article) and "air get polluted" (subject-verb agreement) were not, even tested in isolation. Looked like a coverage gap at first. Follow-up test narrowed it down:

- `"Many person has problem with me."` → only `MANY_NN` fires (flags "Many person").
- `"Many people has problem with me."` → only `PEOPLE_VBZ` fires (flags "has").

So LanguageTool isn't doing general subject-verb agreement parsing — `PEOPLE_VBZ` is a narrow rule keyed to the literal word "people" followed by a singular verb. "person has" doesn't match any built-in pattern, so it's silent; "people has" does. Each match is evaluated against the literal text as typed — LT never reasons about hypothetical corrections ("if error #1 were fixed, would a new error appear"). This is a structural characteristic of self-hosted LanguageTool (a large set of narrow lexical/pattern rules, no full syntactic grammar engine), not a config issue or something fixable on our side without swapping the grammar-checking backend.
