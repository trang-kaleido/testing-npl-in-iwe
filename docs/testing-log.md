## 2026-06-19 â€” recurring closed-vocabulary weakness; discriminative GEC as a possible answer for accuracy

**Bug found â€” sentence-boundary detection is a naive regex, not real segmentation**: both `frontend/src/stall.js` and `frontend/src/ghost.js` find the "current sentence" with `TERMINATOR_RE = /[.?!]/`, scanning backward to the last match. This breaks on abbreviations, decimals, and quoted terminators, and is duplicated independently in both files. Notably, spaCy is already loaded server-side for `/stall` and has a real sentencizer (`doc.sents`), but the boundary is decided client-side via regex before any NLP ever sees the text â€” so a wrong boundary can feed the wrong span into stall's parse. Filed as issue #8.

**Insight â€” the same architectural weakness recurs across stall, ghost text, and accuracy**: in every feature, a real analysis mechanism (spaCy's dependency parse, the ghost-text trie, LanguageTool's pattern engine) correctly diagnoses *that* something applies, then hands back a result from a small, static, closed vocabulary â€” `_VERB_OBJECTS` / `_WORD_COLLOCATES` / `_COHESION_LINKERS` in `backend/main.py` (hand-typed), and the original ghost-text trie before issue #6 (also hand-typed). Issue #6 already proved a fix for this shape: replace the hand-typed list with one mined from a real corpus (STREUSLE), keeping the same trie mechanism but widening its source data. The same move is open for stall's tables, and for using SBERT (currently just a tie-breaking re-ranker) as an open-vocabulary nearest-neighbor retrieval index instead.

**Note for future investigation â€” discriminative (non-generative) GEC for the accuracy gaps**: the 2026-06-18 entry below names two gaps LanguageTool's pattern rules don't cover â€” missing article ("use car") and subject-verb agreement ("air get polluted"). Both are *bounded* edit operations (insert a function word, change a verb form), not open rewrites. Grammarly's own published technique for exactly this, **GECToR** ("Grammatical Error Correction: Tag, Not Rewrite," Omelianchuk et al. 2020), is a sequence *tagger*, not a decoder: a transformer encoder classifies each token into a fixed edit-tag vocabulary (`KEEP`, `DELETE`, `APPEND_the`, `REPLACE_isâ†’are`, verb-inflection tags, etc.), and a deterministic post-process applies the winning tag over a few iterative passes â€” no autoregressive generation. This is structurally closer to ADR 0001's "analyze, don't generate" boundary than a generative GEC model would be, since the model's output is a per-token label from a finite set, not freely composed text â€” though the tag set itself still bakes in specific words (e.g. `APPEND_the`), so whether token-tagging counts as "generation" under ADR 0001 is a real open question worth deciding deliberately before any implementation, not assuming either way.

**Where this still has a ceiling**: GECToR-style tagging only reaches errors inside its fixed tag vocabulary (function words, morphology) â€” open-vocabulary errors (wrong word choice, awkward phrasing) are out of reach for the same reason a generative model would be needed for them in the first place. If pursued, the shape would be augment LT (cheap, explainable, narrow) with a local frozen tagger checkpoint for the structural gaps, plus a merge/rank step â€” not a replacement of either.

**Domain-fit gap (open)**: STREUSLE is hand-annotated Yelp reviews, not academic writing. Checked directly: none of the original IELTS-register phrases ("in terms of", "with regard to", "as opposed to", "a wide range of", etc.) occur in the corpus at all, except "regardless of" / "in spite of" (which didn't pass the â‰Ą2-occurrence cardinality check). So the shipped list proves the extraction *method* works, but doesn't yet cover academic-register ghost text â€” that needs a different, academic-domain corpus with comparable MWE annotation, which wasn't found in this pass.

**Implemented**: edits that overlap an existing diagnostic's span (manual retyping or clicking a suggestion) now trigger an automatic recheck of that sentence, instead of only rechecking when the user retypes the sentence terminator. This closes the "sequential reveal" gap from the entry below: correcting `"Many person has problem with me."` to `"Many people has problem with me."` now immediately surfaces the newly-revealed `PEOPLE_VBZ` flag on "has", without waiting for the user to retype the period.

**Future scope â€” this does not fix LT's coverage gaps**: the recheck loop only helps when a fix changes the literal text enough to match a *different* existing LT rule. It does nothing for errors LT has no rule for at all, in any phrasing â€” e.g. the missing-article ("use car") and subject-verb-agreement ("air get polluted") misses noted below never get caught no matter how many times the sentence is rechecked, since there's no pattern to match in the first place. Closing that gap means changing the grammar-checking backend, not rechecking more.

**Note for future investigation**: Grammarly's pipeline is reportedly a hybrid â€” a layer of fast, narrow, rule/pattern-based checks (similar in spirit to LanguageTool: explainable, low-latency, high-precision) combined with a neural GEC (Grammar Error Correction) model that handles the broader, parsing-dependent corrections rule-based systems miss, with a ranking/merge step reconciling both sets of suggestions. Worth verifying this against primary sources before committing engineering effort, but it maps onto our two earlier options (swap vs. augment): an "augment" path would mean keeping LT for its cheap/explainable matches and adding a GEC model alongside it for the rest, rather than fully replacing LT.

## 2026-06-18 â€” accuracy squiggles (issues #1, #2)

**Setup**: LanguageTool sidecar (`erikvl87/languagetool` Docker image) + FastAPI backend + Vite frontend, run locally per `docs/dev-startup.md`.

**Bug found â€” squiggles didn't clear on fix**: editing the flagged word inside an already-terminated sentence (without retyping the sentence terminator) left the red squiggle in place. Root cause: `diagnosticField` in `frontend/src/editor.js` only ever repositioned diagnostics via `mapPos` on doc changes and merged new diagnostics additively â€” nothing ever invalidated a diagnostic once added, and a clean recheck (zero new matches) didn't dispatch at all. Fixed: edits overlapping a diagnostic's span now drop it immediately, and a recheck replaces (rather than merges with) prior diagnostics in its sentence range, including the zero-match case.

**Insight â€” LanguageTool's rule coverage is pattern-based, not general grammar parsing**: tested `"For example when too many person use car, air get polluted."` â€” only the "many person" agreement error was flagged; "use car" (missing article) and "air get polluted" (subject-verb agreement) were not, even tested in isolation. Looked like a coverage gap at first. Follow-up test narrowed it down:

- `"Many person has problem with me."` â†’ only `MANY_NN` fires (flags "Many person").
- `"Many people has problem with me."` â†’ only `PEOPLE_VBZ` fires (flags "has").

So LanguageTool isn't doing general subject-verb agreement parsing â€” `PEOPLE_VBZ` is a narrow rule keyed to the literal word "people" followed by a singular verb. "person has" doesn't match any built-in pattern, so it's silent; "people has" does. Each match is evaluated against the literal text as typed â€” LT never reasons about hypothetical corrections ("if error #1 were fixed, would a new error appear"). This is a structural characteristic of self-hosted LanguageTool (a large set of narrow lexical/pattern rules, no full syntactic grammar engine), not a config issue or something fixable on our side without swapping the grammar-checking backend.
